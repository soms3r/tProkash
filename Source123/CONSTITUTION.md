# 1. Purpose

This Constitution defines the permanent governing principles of the
tProkash project.

Every architectural decision, document, database schema, API, user
interface, and software component must comply with this Constitution.

If another document conflicts with this Constitution, the Constitution
takes precedence.

------------------------------------------------------------------------

# 2. Vision

Build the most trusted, open, structured, and sustainable publishing
ecosystem for Bangladesh.

------------------------------------------------------------------------

# 3. Mission

-   Organize reliable publishing information.
-   Support authors and publishers.
-   Publish reusable open datasets.
-   Build software on stable architecture.
-   Encourage transparency and interoperability.

------------------------------------------------------------------------

# 4. Core Principles

1.  Accuracy over speed.
2.  Simplicity before complexity.
3.  Documentation before implementation.
4.  Database-first architecture.
5.  Open standards whenever practical.
6.  Reproducible decisions.
7.  Long-term maintainability.
8.  Vendor neutrality.

------------------------------------------------------------------------

# 5. Repository Rules

Every file must:

-   Have a clear purpose.
-   Be placed in the correct directory.
-   Include version information where appropriate.
-   Avoid duplication.
-   Follow naming conventions.

Unused files should not exist.

------------------------------------------------------------------------

# 6. Engineering Principles

-   Prefer stable designs over fashionable technologies.
-   Design for extensibility.
-   Keep backward compatibility whenever feasible.
-   Record major technical decisions using ADRs.
-   Separate documentation, data, and implementation.

------------------------------------------------------------------------

# 7. Documentation Standard

Major documents should include:

-   Title
-   Version
-   Status
-   Dependencies
-   Related documents
-   Change history

Documentation is treated as part of the product.

------------------------------------------------------------------------

# 8. Database Principles

-   Normalize to at least Third Normal Form (3NF) unless justified.
-   Use primary and foreign keys.
-   Use junction tables for many-to-many relationships.
-   Never duplicate authoritative data.
-   Maintain audit timestamps.

------------------------------------------------------------------------

# 9. Data Governance

-   Every published fact should be traceable to a source.
-   Unknown values remain unknown.
-   Verification status must be explicit.
-   No fabricated data.

------------------------------------------------------------------------

# 10. API Principles

-   Resource-oriented design.
-   Stable identifiers.
-   Consistent JSON structure.
-   Versioned APIs.
-   Clear error responses.

------------------------------------------------------------------------

# 11. Security & Privacy

-   Publish only information intended for public use.
-   Respect copyright and licensing.
-   Do not expose confidential or personal information.

------------------------------------------------------------------------

# 12. Versioning

Semantic Versioning:

-   MAJOR: breaking changes
-   MINOR: backward-compatible features
-   PATCH: fixes

------------------------------------------------------------------------

# 13. Decision Process

Major architectural changes require:

1.  Proposal
2.  Discussion
3.  ADR
4.  Approval
5.  Implementation

------------------------------------------------------------------------

# 14. Quality Standard

Project quality is measured by:

-   Accuracy
-   Consistency
-   Maintainability
-   Testability
-   Documentation

------------------------------------------------------------------------

# 15. Amendment

This Constitution may be amended only through a documented proposal and
version update.
